## 테스트 방법 메모

- **인증/세션**: curl로 가입→로그인→`/me`→로그아웃 흐름, 쿠키 저장(`-c`)·재사용(`-b`)으로 세션 유지 확인. 만료 세션은 `sqlite3`로 과거 `expires_at` 값을 가진 행을 직접 INSERT해서 재현.
- **소유권**: 계정 2개(alice/bob)를 만들어 서로의 데이터셋에 대해 조회(성공)·수정(403) 매트릭스를 curl로 전부 확인. 브라우저에서도 같은 계정 쌍으로 UI 컨트롤 노출 여부를 교차 확인.
- **경쟁 상태**: 실제 동시 요청 부하테스트 도구까지는 안 씀 — 코드 경로 리뷰(사전 체크와 INSERT 사이 시간차 확인) + 예외 처리 로직 확인 + 정상/에러 케이스 회귀로 검증.
- **회귀**: Team Day1 완료 후 Day2, Day2 완료 후 회원가입 UX, UX 완료 후 Day3 각 단계 시작 전/후로 이전 curl 스위트를 재실행.
- **UI**: 실제 브라우저에서 로그인 전(로그인/회원가입만 노출)→가입(틀린 초대코드 실패, 맞는 코드 성공)→대시보드 진입→로그아웃→재로그인 흐름과 소유자/비소유자 화면 차이를 콘솔 에러 없이 확인.

## 알려진 한계 (다음에 고려할 것)

- **비밀번호 재설정·계정 복구 없음**: 의도적 결정(PRD §7) — 이메일 등 본인 확인 수단 자체가 없어서 셀프 서비스 복구를 만들 수 없다. 회원가입 화면에 "MVP 데모, 아이디/비번 꼭 기억" 안내 문구로 사용자에게 미리 알림. 실제로 팀이 계속 쓰려면 관리자가 서버 DB를 직접 조작해 복구/재설정해줘야 한다.
- **관리자 페이지 없음**: 초대코드 변경 등 유일한 관리 작업은 터미널 SQL로만 가능(PRD §3.2). 사용자 목록·강제 탈퇴 같은 기능 없음.
- **동시성 실전 부하 미검증**: Pro 때와 동일 — SQLite는 동시 쓰기에 약한데, 실제 동시 다중 요청 부하 테스트는 하지 않음(소규모 팀 학습 규모 전제로 우선순위 낮음 판단).
- **세션 정리(cleanup)는 지연 방식**: 만료된 세션은 그 토큰으로 요청이 들어와야 지워진다 — 아무도 안 쓰는 만료 세션이 DB에 계속 쌓일 순 있음(디스크 용량 자동 정리와 마찬가지로 의도적으로 범위 밖).
