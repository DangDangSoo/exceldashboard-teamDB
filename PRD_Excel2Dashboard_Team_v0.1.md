# Excel2Dashboard — Team 버전 PRD v0.1

## 1. 배경 / 위치

- **3부작 중 3부(마지막).** Basic(완료) → Pro(완료, 영속성) → **Team(이번, 다중 사용자 협업)**.
- **Pro 대비 한 줄 정의:** Pro가 "혼자 업로드한 걸 저장"했다면, Team은 **"여러 사람이 각자 로그인해서 업로드하고, 로그인한 팀원끼리는 서로의 데이터를 볼 수 있다."**
- **Team이 하지 않는 것:** 실시간 협업(동시 편집) — 3부작 계획에 애초에 없던 항목. 세분화된 권한 등급, 조직/그룹 관리, 초대 시스템도 이번엔 범위 밖(§7 참고).
- **시작 방식:** Pro 저장소를 그대로 복사해 새 저장소(`exceldashboard-team`, 가칭)로 시작 — Basic→Pro 전환 때와 동일한 패턴. **이 PRD는 아직 복사 전인 현재(Pro) 저장소에서 작성**하고, 실제 Day 1 구현을 시작하기 전에 저장소 복사를 먼저 진행한다.
- **선행 조사:** [PRD_Excel2Dashboard_Pro_v0.1.md](./PRD_Excel2Dashboard_Pro_v0.1.md), [PRO_DAY0_NOTES.md](./PRO_DAY0_NOTES.md) — Pro의 구조가 그대로 출발점이 된다.

## 2. 확정된 결정 (이번에 사용자와 합의)

| 항목 | 결정 |
|---|---|
| 인증 방식 | **아이디/비밀번호 직접 구현** — OAuth 등 외부 연동 없음 |
| 로그인 유지 방식 | **쿠키 기반 서버 세션** |
| 데이터셋 공유 범위 | **로그인한 모든 사용자에게 전체 공개** — 별도 공유 설정 없이, 로그인만 하면 팀원 전체의 데이터셋을 조회 가능 |
| 권한 수준 | **단순** — 소유자만 수정·삭제·태그변경·분석저장 가능, 나머지는 조회만 |
| 저장소 시작 방식 | Pro 저장소 복사, **Pro의 기존 데이터는 이어받지 않고 빈 DB로 시작** |
| 비밀번호 저장 | **`bcrypt`로 해싱**(평문 저장 금지) — 구현 세부사항이라 별도 질문 없이 결정 |
| 회원가입 범위 | **고정 초대코드 필요** — 회원가입 시 초대코드를 함께 입력해야 계정이 만들어짐. 초대코드는 DB `settings` 테이블에 값 하나로 저장되고, 바꿀 땐 관리자 페이지 없이 `sqlite3 data/app.db "UPDATE settings SET value='새코드' WHERE key='invite_code'"`처럼 터미널에서 직접 변경(§3.2 참고) |
| 관리자 페이지 | **만들지 않음** — 초대코드 관리는 터미널로 충분하다고 판단(§7 참고). 나중에 필요해지면 `users.is_admin` 컬럼 + 관리자 전용 화면을 얹는 식으로 확장 가능(지금 구조를 갈아엎지 않아도 됨) |

## 3. 스택 / 아키텍처

- **백엔드:** Pro와 동일(FastAPI + pandas + matplotlib + SQLite) + 인증 계층 추가.
- **세션:** SQLite `sessions` 테이블(세션 토큰 → user_id, 만료시각)에 저장 — Pro가 이미 "DB가 진짜 저장소"라는 원칙을 세워뒀으므로 세션도 같은 원칙을 따름(서버 재시작해도 로그인 유지됨). 쿠키엔 추측 불가능한 랜덤 토큰만 담고, 매 요청마다 DB에서 유효성을 확인한다(JWT처럼 자체 서명 방식은 안 씀 — 토큰 무효화가 DB 행 삭제 한 번으로 끝나 더 단순함).
- **프론트:** 로그인/회원가입 화면 추가. 로그인 전에는 업로드·데이터셋 화면 자체가 안 보임.

### 3.1 이음새 원칙 승계

- `app/core/*.py` — Team에서도 무수정.
- 기존 `Dataset`/`DatasetSummary` 모델에 `owner_username` 필드만 추가(필드 추가 허용, 기존 필드 의미 변경 없음 — 이음새 원칙 계속 준수).
- Pro가 만든 dataset uuid 체계는 그대로 두고 `owner_id`만 얹는다.

### 3.2 초대코드 관리 (관리자 페이지 대신)

회원가입을 막는 방법으로 "관리자 페이지"까지는 과하다고 판단해 더 가벼운 방식을 쓴다:
- `settings` 테이블에 `key='invite_code'` 행 하나로 초대코드를 저장(§4 참고). 서버 최초 기동 시 기본값(예: `CHANGEME`)으로 자동 생성.
- 바꾸고 싶으면 서버에 접근 권한이 있는 사람이 `sqlite3 data/app.db "UPDATE settings SET value='새코드' WHERE key='invite_code'"`로 직접 변경 — 별도 화면·API 없음.
- 회원가입 API가 요청받은 코드와 `settings` 값을 비교해서 다르면 400.
- 사용자별 개별 초대코드, 1회용 코드, 만료 등은 만들지 않는다(§7). 필요해지면 `users.is_admin` + 관리자 화면으로 확장하되, 지금 구조를 갈아엎을 필요는 없다.

## 4. 데이터 모델

```
users 테이블
  id              TEXT PRIMARY KEY (uuid)
  username        TEXT UNIQUE NOT NULL
  password_hash   TEXT NOT NULL
  created_at      TEXT

sessions 테이블
  token           TEXT PRIMARY KEY   -- 쿠키에 담기는 랜덤 토큰
  user_id         TEXT NOT NULL REFERENCES users(id)
  created_at      TEXT
  expires_at      TEXT

settings 테이블
  key             TEXT PRIMARY KEY   -- 예: 'invite_code'
  value           TEXT NOT NULL

datasets 테이블 (Pro 스키마에 컬럼 추가)
  owner_id        TEXT NOT NULL REFERENCES users(id)
```

- Team은 **새 저장소 + 빈 DB**로 시작하므로, Pro의 `datasets` 스키마에 `owner_id`를 나중에 `ALTER TABLE`로 추가할 필요 없이 **처음부터 포함해서** 만들면 된다. 마이그레이션 이슈 없음.

## 5. 기능 요구사항

### 5.1 회원가입 · 로그인 · 로그아웃
- `POST /api/auth/register` `{username, password, invite_code}` — `invite_code`를 `settings` 값과 대조(틀리면 400) → 아이디 중복 체크 → 비밀번호 `bcrypt` 해싱 후 저장.
- `POST /api/auth/login` `{username, password}` — 검증 후 `sessions`에 토큰 발급, 쿠키로 내려줌.
- `POST /api/auth/logout` — 현재 세션을 `sessions`에서 삭제 + 쿠키 제거.
- `GET /api/auth/me` — 로그인 여부와 현재 사용자 정보 확인(프론트가 첫 로드 시 호출해서 로그인 화면 vs 대시보드 화면 결정).

### 5.2 접근 제어
- 로그인하지 않은 상태로 `/api/datasets/*`, `/api/upload` 등에 접근하면 401.
- 조회(GET: 목록·상세·통계·차트생성·집계·피벗·분석재현)는 로그인한 사용자 누구나 가능(§2 "전체 공개" 결정).
- 변경(PUT 태그·열타입, DELETE 데이터셋·분석, POST 분석저장)은 **본인 소유(`owner_id` 일치)만 가능**, 아니면 403.
- 업로드(`POST /api/upload`)는 로그인한 사용자 본인 소유로 생성됨(`owner_id` = 현재 로그인한 사용자).

### 5.3 데이터셋에 소유자 표시
- 데이터셋 목록·상세 화면에 업로드한 사용자 이름을 표시해 "누구 데이터인지" 구분 가능하게 한다.

### 5.4 프론트엔드
- 로그인 전: 로그인/회원가입 폼만 보이고 나머지 화면은 숨김. 회원가입 폼에는 아이디·비밀번호 외에 **초대코드 입력칸**이 추가로 있음.
- 로그인 후: 기존 Pro 화면 그대로 + 우측 상단에 "OO님 · 로그아웃" 표시. 본인 소유가 아닌 데이터셋 행에는 수정/삭제 버튼이 아예 안 보이거나 비활성화(어차피 서버에서도 403으로 막히지만, 눌러보고 에러 보는 것보다 UI에서 먼저 걸러주는 게 매끄러움).

## 6. 비기능 / 제약

- **세션 만료:** 발급 후 7일. 만료되면 재로그인 필요.
- **동시성:** SQLite 특성상 동시 쓰기에 약한 건 Pro 때와 동일 — 실사용 트래픽이 적은 학습/소규모 팀 규모라는 전제로 이번에도 보류.
- **비밀번호 정책:** 최소 길이 등 강제 규칙은 두지 않는다(학습 목적 범위 밖, 필요시 나중에 추가).

## 7. 만들지 말 것 (Team 범위 밖)

- **실시간 협업(동시 편집)** — Basic PRD 때부터 3부작 어디에도 계획에 없던 항목.
- **세분화된 권한(편집자 등급 등)** — 이번엔 소유자/조회자 2단계만(§2 결정).
- **조직/그룹 관리, 사용자별 개별 초대 시스템** — 고정 초대코드 하나로 통일(§2, §3.2 결정). 1회용 코드·사용자별 코드·코드 만료는 없음.
- **개별 데이터셋 단위 공유 설정(비공개/특정인 공개)** — "전체 공개" 하나로 통일(§2 결정).
- **관리자 페이지·관리자 역할(`is_admin`)** — 초대코드 하나 관리하자고 화면·권한 체계를 새로 만드는 건 과함. 터미널에서 DB 값 직접 수정으로 대체(§3.2). 나중에 필요해지면 얹을 수 있는 구조로만 남겨둠.

## 8. 단계 분할

- **Day 1 — 계정·인증:** `users`/`sessions` 테이블 + 회원가입/로그인/로그아웃/`me` API + 프론트 로그인·회원가입 화면 + 기존 API 전체에 로그인 필수화(401 처리). *(만들지 말 것: 소유권 검증 — 아직 모든 로그인 사용자가 뭐든 수정 가능한 과도기 상태로 둠.)*
- **Day 2 — 소유권·접근 제어:** `datasets.owner_id` + 업로드 시 소유자 기록 + 변경계열 API에 소유자 검증(403) + 목록/상세에 소유자 이름 표시 + 프론트에서 본인 것만 수정/삭제 버튼 노출.
- **Day 3 — 견고화·문서:** 권한 엣지케이스(타인 데이터 수정 시도, 세션 만료 처리, 중복 아이디 가입 시도 등) + README·자체평가 갱신.

## 9. 완료 기준

1. **가입·로그인:** 올바른 초대코드를 입력해야 회원가입이 되고(틀리면 명확한 에러), 회원가입한 사용자만 로그인할 수 있으며, 로그인해야 데이터셋 화면에 접근할 수 있다.
2. **전체 공개 조회:** 로그인한 팀원은 자신이 올리지 않은 데이터셋도 목록·상세·통계·차트를 조회할 수 있다.
3. **소유권 보호:** 본인이 올리지 않은 데이터셋/분석에 대한 수정·삭제 시도는 403으로 막힌다(API 레벨에서도, UI에서도).
4. **세션 지속성:** 서버를 재시작해도(만료 전까지) 로그인 세션이 유지된다.
5. **회귀 없음:** Pro의 업로드·태그·재호출·분석저장·삭제 흐름이 "로그인한 사용자" 전제 하에서 예전과 동일하게 동작한다.
6. **문서:** README에 Team 기능·API·실행법 반영, 자체평가 갱신.
