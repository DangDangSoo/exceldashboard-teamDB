# Excel2Dashboard — Pro 버전 PRD v0.1

## 1. 배경 / 위치

- **3부작 중 2부.** Basic(완료) → **Pro(이번, 영속성)** → Team(다중 사용자 협업). 지난 것을 버리지 않고 한 겹만 얹는다.
- **Basic 대비 한 줄 정의:** Basic은 "업로드 → 분석 → 새로고침하면 사라짐"이었다면, Pro는 **"업로드한 파일과 만든 분석을 저장해두고 나중에 다시 불러올 수 있다."**
- **Pro가 하지 않는 것:** 로그인·계정·다중 사용자·공유·권한. 지금도 사용자는 한 명이고, 저장된 데이터는 이 서버를 쓰는 누구나 볼 수 있는 상태로 둔다 (접근 제어는 Team의 몫).
- **선행 조사:** [PRO_DAY0_NOTES.md](./PRO_DAY0_NOTES.md)에 Basic 코드 구조 분석과 확장 지점이 이미 정리되어 있음 — 이 PRD는 그 위에서 결정만 채운 것.

## 2. 확정된 결정 (이번에 사용자와 합의)

| 항목 | 결정 |
|---|---|
| DB 종류 | **SQLite** — 파일 하나, 별도 서버 불필요, 개인용 규모에 적합 |
| 원본 파일 저장 방식 | **디스크에 파일로 저장** (`uploads/{dataset_id}/{filename}`), DB에는 경로만 기록 |
| 데이터 모델 | **프로젝트 대신 태그(#tag) 기반 모아보기** — 데이터셋은 여전히 업로드 단위로 독립 관리하되, 데이터셋마다 태그를 여러 개 붙여 나중에 필터링해서 모아볼 수 있게 함 |
| Team 대비 여부 | **지금은 안 함** — owner/project 컬럼을 미리 만들지 않는다 (YAGNI) |
| DB 접근 방식 | **`sqlite3` 직접 SQL** — ORM(SQLAlchemy) 미사용, 학습 목적상 SQL이 직접 보이는 쪽 선택 |
| 데이터셋 제목 | **파일명 그대로 표시** — 커스텀 제목 입력 UI 없음 |
| 태그 입력 방식 | **자유 텍스트 입력** — 자동완성 없음, 콤마로 구분해 입력 |
| `saved_analyses` 테이블 구조 | **단일 테이블**(kind 컬럼으로 구분) — 종류별 3테이블로 나누지 않음 |
| 태그 정규화 | **trim + 소문자 통일 적용** — 앞뒤 공백 제거, 영문은 소문자로 맞춰 중복 방지 (한글은 대소문자 구분 없으니 영향 없음) |
| 재호출 시 원본 파일 없음 | **404 + "원본 파일을 찾을 수 없습니다. 다시 업로드해주세요." 메시지** |

## 3. 스택 / 아키텍처

- **백엔드:** Basic과 동일 (Python + FastAPI + pandas + matplotlib) + **SQLite** 저장 계층 추가.
- **DB 접근:** 표준 라이브러리 `sqlite3`로 직접 SQL 작성 (ORM 미사용, §2 결정).
- **파일 저장:** 서버 로컬 디스크의 `uploads/` 폴더 (`.gitignore`에 이미 등록됨).
- **프론트:** Basic의 바닐라 HTML/JS 유지 + "내 데이터셋" 목록 화면, "분석 저장" 버튼 추가.

### 3.1 이음새 3원칙 — Pro에서도 반드시 지킬 것

[PRO_DAY0_NOTES.md](./PRO_DAY0_NOTES.md:64)에서 이미 검증됨:

1. **`app/core/*.py`는 손대지 않는다.** FastAPI·DB를 모르는 순수 함수 그대로 재사용. 저장은 라우터 계층에서만 처리.
2. **기존 JSON 계약(`Dataset`/`ChartSpec`/`AggregateSpec`/`PivotSpec`) 필드를 유지한다.** 필드 추가는 가능, 기존 필드 의미 변경 금지 — 이 구조 그대로 DB 레코드의 payload가 된다.
3. **Basic 기능은 회귀 없이 그대로 동작해야 한다.** 저장 버튼을 누르지 않는 한, 지금처럼 "업로드 → 분석 → PNG 다운로드"가 세션 1회성으로도 계속 작동해야 한다. 저장은 **선택적으로 추가되는 기능**이지, 기존 흐름을 저장 중심으로 뒤집는 게 아니다.

### 3.2 데이터 모델 (초안)

```
datasets 테이블
  id              TEXT PRIMARY KEY (uuid, Basic의 dataset_id 그대로 승격)
  filename        TEXT
  file_path       TEXT   -- uploads/{id}/{filename}
  row_count       INTEGER
  col_count       INTEGER
  columns_json    TEXT   -- ColumnMeta 리스트 (pydantic .model_dump_json())
  uploaded_at     TEXT   -- ISO 8601

dataset_tags 테이블
  dataset_id      TEXT   -- FK -> datasets.id
  tag             TEXT   -- 예: '매출', '2024년' ('#'는 화면 표시용 기호일 뿐, DB엔 안 붙임)
  PRIMARY KEY (dataset_id, tag)

saved_analyses 테이블
  id              TEXT PRIMARY KEY (uuid)
  dataset_id      TEXT   -- FK -> datasets.id
  kind            TEXT   -- 'chart' | 'aggregate' | 'pivot'
  title           TEXT   -- 사용자가 붙인 이름, 없으면 자동 생성(예: "히스토그램 - 매출액")
  spec_json       TEXT   -- ChartSpec/AggregateSpec/PivotSpec 중 하나 (.model_dump_json())
  created_at      TEXT
```

- `saved_analyses`를 종류별로 3개 테이블로 나누지 않고 하나로 합친 이유: 세 Spec 모두 이미 pydantic JSON이라 그대로 문자열로 넣을 수 있고, "저장된 분석 목록"을 보여줄 때 테이블 하나만 조회하면 되기 때문 (§2 결정).
- `datasets.columns_json`은 Basic의 `Dataset.columns`를 그대로 저장한 것. 재호출 시 원본 파일을 다시 `parse_file`로 읽고, 저장해둔 `columns_json`을 타입 정보로 재사용(재판별 불필요, 수동 교정 이력도 보존됨).
- `dataset_tags`를 `datasets` 안에 JSON 배열 컬럼으로 넣지 않고 별도 테이블로 분리한 이유: 태그로 필터링(`WHERE tag = '매출'`)하는 게 핵심 기능이라, SQL로 바로 검색 가능한 구조가 유리하기 때문. 데이터셋 하나가 여러 태그를 가질 수 있는 다대다 구조라 프로젝트(1:N)보다 유연함.

## 4. 기능 요구사항

### 4.1 업로드 시 영속 저장 (기존 흐름 확장)
- `POST /api/upload`는 기존과 동일하게 동작하되, 추가로:
  - 원본 파일을 `uploads/{dataset_id}/{filename}`에 저장.
  - `datasets` 테이블에 메타데이터 1행 기록.
- 세션 메모리(`session_store`)는 "지금 열려 있는 데이터셋의 DataFrame 캐시" 용도로 유지 — 매 요청마다 디스크에서 다시 읽지 않기 위함. DB/디스크는 "진짜 저장소", 메모리는 캐시라는 역할 분리.

### 4.2 태그 붙이기 · 모아보기
- 업로드 시(또는 업로드 후 언제든) 데이터셋에 태그를 자유롭게 붙일 수 있음. 예: `#2024년`, `#매출`.
- `POST /api/upload` 요청에 선택적 `tags` 파라미터 추가(콤마 구분 문자열 또는 배열) — 없으면 태그 없이 저장.
- 신규 `PUT /api/datasets/{id}/tags` — 이미 저장된 데이터셋의 태그를 나중에 추가/수정.
- `GET /api/datasets` 목록 조회에 `?tag=매출` 필터 옵션 추가 — 같은 태그가 붙은 데이터셋만 모아 보기.
- 저장 전 정규화: 앞뒤 공백 제거(trim) + 영문 소문자 통일 후 저장/비교 (예: `Sales`, `sales `, `sales` 모두 같은 태그로 취급, §2 결정).
- 프론트: 업로드 폼에 태그 입력칸(콤마로 구분), 데이터셋 목록 화면에 태그 필터 칩(chip) 표시. 자동완성은 넣지 않음(§2 결정).

### 4.3 데이터셋 목록 · 재호출
- 신규 `GET /api/datasets` — 저장된 데이터셋 목록(파일명, 업로드일, 행/열 수) 반환.
- 신규 `GET /api/datasets/{id}` (또는 `/reload`) — 이미 세션 메모리에 없으면 `file_path`에서 원본을 다시 읽어(`parse_file` 재사용) `session_store`에 올리고, 저장해둔 `columns_json`(수동 교정 이력 포함)을 복원 → 이후는 Basic과 동일한 화면(통계·차트·집계) 그대로 사용.
- `file_path`가 가리키는 파일이 디스크에 없으면 404 + "원본 파일을 찾을 수 없습니다. 다시 업로드해주세요." 메시지로 응답 (§2 결정).
- 프론트: 첫 화면에 "새로 업로드" 아니면 "저장된 데이터셋에서 이어하기" 목록 표시.

### 4.4 분석 결과 저장 · 재현
- 차트/집계/피벗 빌더에 **"저장" 버튼** 추가 (기존 "다운로드" 버튼 옆).
- 신규 `POST /api/datasets/{id}/analyses` — 현재 spec(JSON)을 `saved_analyses`에 기록.
- 신규 `GET /api/datasets/{id}/analyses` — 저장된 분석 목록.
- 저장된 분석을 클릭하면 같은 spec으로 기존 `render_chart`/`render_aggregate_chart`/`render_pivot_chart`를 다시 호출해 PNG 재생성 (PNG 자체는 저장하지 않고 spec만 저장 — 항상 최신 데이터 기준으로 재계산됨).

### 4.5 삭제
- 신규 `DELETE /api/datasets/{id}` — DB 행 + 디스크 파일 + 딸린 `saved_analyses` 함께 삭제.
- 신규 `DELETE /api/datasets/{id}/analyses/{analysis_id}` — 저장된 분석 1건 삭제.
- 실수 방지용 확인창은 프론트에서 처리(간단한 `confirm()` 수준으로 충분).

## 5. 비기능 / 제약

- **Basic 제약 전부 승계:** 파일 크기(10MB)/행 수(5만) 상한, CSV 인코딩 자동감지, 지저분한 데이터 방어, 조용한 실패 금지 — 그대로 유지.
- **디스크 용량 관리는 범위 밖.** 업로드 파일이 계속 쌓여도 자동 정리는 하지 않는다 (개인용 학습 규모 전제). 필요 시 수동 삭제(§4.5)로 충분.
- **DB 파일 위치:** 프로젝트 루트의 `data/app.db` (신규 `.gitignore` 항목 추가 필요).
- **동시성:** SQLite는 동시 쓰기에 약하지만, 단일 사용자 전제이므로 문제 없음 (Team 단계에서 재검토).

## 6. 만들지 말 것 (Pro 범위 밖)

- 로그인 · 계정 · 다중 사용자 · 공유 · 권한 → **Team**
- 계층형 "프로젝트" 그릇(생성/이름변경/소속 데이터셋 관리) → 태그(§4.2)로 대체. 데이터셋이 여러 그룹에 동시에 속할 수 있어 프로젝트보다 이 앱 규모에 맞음.
- 태그 자체의 관리 화면(태그 목록/이름변경/삭제 CRUD) → 태그는 데이터셋에 붙이고 떼는 라벨일 뿐, 독립된 엔티티로 관리하지 않음.
- 저장된 PNG 자체를 캐싱/보관 → 항상 spec으로부터 재생성 (원본 데이터가 바뀔 일이 없으니 캐싱은 과함)
- 다중 시트, SVG 내보내기, 실시간 협업, 예측/머신러닝 — Basic PRD와 동일하게 3부작 어디에도 계획 없음

## 8. 완료 기준

1. **영속성:** 업로드한 데이터셋이 서버를 재시작해도 목록에 남아있고, 다시 불러오면 Basic과 동일하게 통계·차트가 재현된다.
2. **재호출 정확성:** 수동으로 교정한 열 타입이 재호출 후에도 유지된다.
3. **태그 모아보기:** 데이터셋에 태그를 붙일 수 있고, 태그로 목록을 필터링하면 해당 태그가 붙은 데이터셋만 정확히 모여서 보인다.
4. **분석 저장·재현:** 저장한 차트/집계/피벗이 목록에 남고, 다시 열면 저장 당시와 동일한 결과가 나온다.
5. **삭제:** 데이터셋/분석 삭제 시 DB 행과 디스크 파일이 함께 정리되고, 목록에서 사라진다.
6. **회귀 없음:** 저장 버튼을 쓰지 않는 기존 Basic 흐름(업로드→분석→PNG 다운로드)이 그대로 동작한다.
7. **문서:** README에 Pro 기능·API·실행법 반영, 자체평가 갱신.
